# AssetID System Guide

## Overview

The AssetID system is Show-Build's centralized asset identification and tracking system. Every content asset (episodes, segments, cues, graphics, media files) gets a unique AssetID that enables precise tracking, cross-referencing, and workflow management throughout the broadcast production pipeline.

## What is an AssetID?

An AssetID is a **unique alphanumeric identifier** assigned to every asset in the Show-Build ecosystem. AssetIDs serve as:

- **Universal References**: Single identifier for any asset across all systems
- **Database Keys**: Primary identifiers for database records and relationships  
- **Cross-Platform Tracking**: Links between Obsidian files, backend database, and external systems
- **Audit Trail**: Enables tracking of asset creation, modification, and relationships
- **API Integration**: Standard identifiers for all API operations

## AssetID Types

### Server AssetIDs (Recommended)
- **Generated by**: Show-Build AssetID service (`/newAssetID` endpoint)
- **Format**: 10+ character uppercase alphanumeric strings
- **Examples**: `EPSMEEJ32D9UFK5BU`, `SEGABC123DEF456`, `CUEGFX789XYZ123`
- **Database**: Stored in `asset_id_registry` table with full metadata
- **Features**: 
  - Guaranteed global uniqueness
  - Database-backed with audit trail
  - Parent-child relationship tracking
  - Cross-system compatibility

### Local AssetIDs (Legacy/Temporary)
- **Generated by**: Local systems (Obsidian, manual entry)
- **Format**: Various patterns, typically lowercase with underscores
- **Examples**: `gfx_20250816_001`, `LOCAL-ABC123`, `sot_2024_interview`
- **Database**: Not registered in `asset_id_registry`
- **Usage**: Temporary identifiers, should be converted to server AssetIDs

## AssetID Assignment Process

### 1. Automatic Assignment (Recommended)

**When**: During content creation via Show-Build API or unified endpoints

```python
# Example: Creating new episode
POST /newAssetID
{
  "asset_type": "episode",
  "episode_number": 236,
  "slug": "sunday-show-236"
}

# Response: 
{
  "asset_id": "EPSMEEJ32D9UFK5BU",
  "status": "created",
  "database_record_id": 123
}
```

### 2. Manual Assignment (Obsidian Plugin)

**When**: Working in Obsidian with cue-manager plugin

- **Alt+9**: Convert local AssetID to server AssetID in current context
- **Context Detection**: Automatically detects episode/segment/cue context
- **Database Integration**: Creates corresponding database records

### 3. Bulk Assignment (Backend API)

**When**: Converting existing content or bulk operations

```bash
# Convert all AssetIDs in episode
POST /api/convertID/all/0236

# Convert specific segment
POST /api/convertID/segment/0236/interview-segment

# Convert cue blocks in segment  
POST /api/convertID/cues/0236/interview-segment
```

## Database Integration

### Core Tables

#### `asset_id_registry` (Primary Registry)
```sql
asset_id          VARCHAR(50)  -- The actual AssetID
entity_type       VARCHAR(50)  -- episode, segment, cue, etc.
entity_id         VARCHAR(50)  -- Internal ID reference
status            VARCHAR(20)  -- active, inactive, archived
created_at        TIMESTAMP    -- Creation timestamp
created_by        VARCHAR(100) -- User/system that created it
reason            VARCHAR(255) -- Why it was created
linked_to         JSON         -- Parent-child relationships
context           JSON         -- Additional metadata
```

#### Content Tables with AssetID Fields
- **episodes**: `asset_id` field links to registry
- **segments**: `asset_id` field links to registry  
- **cues**: `asset_id` field links to registry
- **rundown_items**: `asset_id` field links to registry
- **asset_links**: Parent-child relationship tracking

### Database Verification

The system includes database checks to verify AssetID authenticity:

```python
def _is_server_asset_id(self, asset_id: str) -> bool:
    # 1. Pattern matching (uppercase, 10+ chars)
    # 2. Local pattern exclusion (underscores, dates)
    # 3. Database registry lookup (most reliable)
    registry_entry = db.query(AssetIDRegistry).filter(
        AssetIDRegistry.asset_id == asset_id
    ).first()
    return registry_entry is not None
```

## API Endpoints

### AssetID Generation

#### `POST /newAssetID` - Unified AssetID Creation
Creates new server AssetID with database record and relationships.

**Request:**
```json
{
  "asset_type": "episode|segment|cue|ad|promo|cta",
  "slug": "descriptive-name",
  "parent_asset_id": "PARENT123ABC456",  // Optional
  "cue_type": "SOT|GFX|FSQ|VO|NET",     // For cues
  "episode_number": 236,                 // For episodes
  "context": {"key": "value"}            // Additional metadata
}
```

**Response:**
```json
{
  "asset_id": "NEWASSET123ABC456",
  "asset_type": "episode",
  "database_record_id": 789,
  "parent_asset_id": "PARENT123ABC456",
  "created_by": "api_user",
  "created_at": "2025-08-16T13:45:00Z",
  "status": "created"
}
```

### AssetID Conversion

#### `GET /api/scanAssetIDs` - Asset Discovery & Analysis
Scans filesystem for all AssetIDs and identifies conversion needs.

**Parameters:**
- `episode`: Optional episode number filter

**Response:**
```json
{
  "success": true,
  "message": "Scanned 45 AssetIDs: 23 server, 22 local, 22 need conversion",
  "total_assets_found": 45,
  "server_assets_count": 23,
  "local_assets_count": 22,
  "conversion_needed_count": 22,
  "assets": [
    {
      "asset_id": "gfx_20250816_001",
      "file_path": "/home/episodes/0236/rundown/intro.md",
      "context_type": "cue",
      "episode_number": "0236",
      "segment_name": "intro",
      "cue_type": "GFX",
      "is_server_asset": false,
      "needs_conversion": true,
      "conversion_api_call": "POST /api/convertID/cues/0236/intro"
    }
  ],
  "suggested_conversions": [
    "POST /api/convertID/episode/0236",
    "POST /api/convertID/segment/0236/intro",
    "POST /api/convertID/cues/0236/intro"
  ]
}
```

#### `POST /api/convertID/episode/{episode_number}` - Episode Conversion
Converts episode `info.md` AssetID from local to server-registered.

#### `POST /api/convertID/segment/{episode}/{segment}` - Segment Conversion  
Converts segment frontmatter AssetID from local to server-registered.

#### `POST /api/convertID/cues/{episode}/{segment}` - Cue Conversion
Converts all cue block AssetIDs in a segment file.

#### `POST /api/convertID/all/{episode}` - Bulk Conversion
Converts ALL AssetIDs in an episode (episode + segments + cues).

### AssetID Removal

#### `POST /removeAssetID` - Asset Removal
Removes AssetID from registry and associated database records.

**Request:**
```json
{
  "asset_id": "ASSET123ABC456",        // Direct removal
  // OR context-based removal:
  "episode_number": 236,
  "context_type": "episode"
}
```

## File System Integration

### Frontmatter Integration

AssetIDs are stored in YAML frontmatter of markdown files:

```yaml
---
AssetID: EPSMEEJ32D9UFK5BU
episode_number: 0236
title: "Sunday Show Episode 236"
status: draft
---
```

### Cue Block Integration

AssetIDs are embedded in cue blocks within segment files:

```markdown
<!-- Begin Cue -->
[Type: GFX]
[AssetID: CUEGFX789XYZ123]
[Slug: vigilante-tube]
[Description: Lower third graphic]
[MediaURL: ../assets/graphics/vigilante-tube.png]
<!-- End Cue -->
```

### System Messages

The system adds timestamped messages to track AssetID operations:

```yaml
---
system_message: 2025-08-16T17:25:44.091Z: Generated new server AssetID EPSMEEJ32D9UFK5BU
AssetID: EPSMEEJ32D9UFK5BU
---
```

## Parent-Child Relationships

AssetIDs maintain hierarchical relationships for content organization:

### Hierarchy Structure
```
Episode (EPSMEEJ32D9UFK5BU)
├── Segment 1 (SEGABC123DEF456)
│   ├── Cue 1 (CUEGFX789XYZ123)
│   └── Cue 2 (CUESOT456GHI789)
└── Segment 2 (SEGDEF456GHI789)
    └── Cue 3 (CUEVO123JKL456)
```

### Database Tracking
```sql
-- asset_links table
source_asset_id: EPSMEEJ32D9UFK5BU  (Episode)
target_asset_id: SEGABC123DEF456    (Segment)
link_type: parent_child
meta_data: {"child_type": "segment", "relationship": "parent_to_child"}
```

## Obsidian Plugin Integration

### Key Commands

| Hotkey | Command | Function |
|--------|---------|----------|
| `Alt+9` | Ensure AssetID | Convert local AssetID to server AssetID |
| `Shift+Alt+0` | Remove AssetID | Delete AssetID from database |

### Context Detection

The plugin automatically detects context based on cursor position:

1. **Episode Context**: Cursor in `info.md` file
2. **Segment Context**: Cursor in segment frontmatter  
3. **Cue Context**: Cursor within `<!-- Begin Cue -->` blocks

### Authentication

Plugin uses API key authentication via `X-API-Key` header for backend communication.

## Best Practices

### 1. Always Use Server AssetIDs
- **Avoid** local AssetIDs for new content
- **Convert** existing local AssetIDs to server AssetIDs
- **Use** unified `/newAssetID` endpoint for all new assets

### 2. Maintain Parent-Child Relationships
- **Link** segments to episodes via `parent_asset_id`
- **Link** cues to segments via `parent_asset_id`
- **Verify** relationships using AssetID scanner

### 3. Regular Scanning and Cleanup
- **Run** `/api/scanAssetIDs` periodically to identify conversion needs
- **Convert** local AssetIDs before they proliferate
- **Monitor** database integrity with AssetID registry

### 4. API Integration
- **Use** proper authentication (API keys or JWT)
- **Check** database verification for AssetID authenticity
- **Handle** errors gracefully with proper status codes

## Troubleshooting

### Common Issues

#### "AssetID already exists" Error
- **Cause**: Attempting to create duplicate AssetID
- **Solution**: Use existing AssetID or check database for conflicts

#### "AssetID not found in registry" Error  
- **Cause**: Attempting to use local/invalid AssetID
- **Solution**: Convert to server AssetID using conversion endpoints

#### Parent-Child Link Failures
- **Cause**: Parent AssetID doesn't exist or invalid
- **Solution**: Verify parent exists in database, create if necessary

#### Database Sync Issues
- **Cause**: AssetID exists in files but not in database
- **Solution**: Use conversion endpoints to sync filesystem with database

### Debugging Tools

#### AssetID Scanner

# Comprehensive scan

#### Database Queries
```sql
-- Check AssetID registry
SELECT * FROM asset_id_registry WHERE asset_id = 'ASSET123ABC456';

-- Check parent-child relationships
SELECT * FROM asset_links WHERE source_asset_id = 'PARENT123' OR target_asset_id = 'CHILD456';

-- Find orphaned AssetIDs
SELECT * FROM asset_id_registry 
WHERE asset_id NOT IN (SELECT asset_id FROM episodes WHERE asset_id IS NOT NULL);
```

## Security Considerations

### Authentication Required
- All AssetID endpoints require authentication (API key or JWT)
- API keys stored securely with proper RBAC permissions

### Data Integrity
- Database constraints prevent duplicate AssetIDs
- Foreign key relationships maintain referential integrity
- Audit trails track all AssetID operations

### File System Protection
- Docker user permissions (1000:1001) control file access
- Backup systems protect against data loss during conversions

## Performance Considerations

### Batch Operations
- Use bulk conversion endpoints for multiple assets
- Scanner endpoint optimized for large episode collections  
- Database indexing on AssetID fields for fast lookups

### Caching Strategy
- AssetID registry cached for frequent lookups
- Database connection pooling for concurrent requests
- File system operations batched when possible
