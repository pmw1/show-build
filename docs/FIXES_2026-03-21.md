## 1. Script Mode paragraphs display with excess blank lines

**Problem:** In Script Mode, each paragraph rendered with ~7 extra blank line numbers below the actual text content. The contenteditable div had no `flex` or width constraint in its flex container, causing unreliable `scrollHeight` measurement, and the line count cache was never cleared between item switches so stale values persisted.

**Fix:** Added `flex: 1; min-width: 0;` to `.speaker-contenteditable` for proper width constraint. Replaced the fragile multi-timeout `scrollHeight` measurement (0ms/100ms/300ms after mount) with a `ResizeObserver` on each contenteditable that recalculates line count from `contentRect.height` whenever the element actually resizes — always accurate, fires after layout settles. Also clear `segmentLineCounts = {}` in the `scriptSegments` watcher so stale cached line counts don't carry over between items.

## 2. ESC key in modals closes underlying segment instead of just the modal

**Problem:** Pressing ESC while a modal (e.g., GfxModal) was open would close the modal AND deselect/exit the current segment, potentially losing unsaved work. Vuetify's v-dialog handles ESC internally and sets the v-model to `false` before ContentEditor's document-level keydown handler fires, making all reactive modal flags false — so the handler thought no modal was open and proceeded to exit editing mode.

**Fix:** Added a `lastModalCloseTime` timestamp that is set (via Vue watchers on every `show*Modal` flag) whenever any modal transitions from open to closed. The ESC handler now checks `Date.now() - lastModalCloseTime < 300` as a final fallback in its `hasModalOpen` check, catching the race condition where Vuetify already dismissed the dialog in the same event loop tick.

## 3. Alt+G hotkey pre-populates GfxModal with stale whiteboard data

**Problem:** After using the whiteboard "+" button to insert an X post (which sets `whiteboardPrefillData`), pressing Alt+G to open a fresh GfxModal would show the previous X post's data pre-populated instead of a blank form.

**Fix:** Added `this.whiteboardPrefillData = null` in `handleShowGfxModal()` alongside the existing `editingGfxCueData = null` clear, so the hotkey path always opens a clean modal.

## 4. GFX-XPOST social tab submit blocked

**Problem:** The GfxModal's submit handler had a guard clause (`if (activeTab.value !== 'gfx')`) that showed a "Not Implemented" dialog for all non-GFX tabs, including the social tab which was fully built and functional.

**Fix:** Added a check for `activeTab.value === 'social'` before the guard, routing social tab submissions to the existing `handleSocialInsert()` method which already builds the complete xpost cue data.

## 5. Twitter API v2 not extracting views or bookmarks

**Problem:** The `fetch_tweet_from_api()` function in `social_media.py` requested `public_metrics` from the Twitter API (which includes `impression_count` and `bookmark_count`) but only extracted `like_count`, `retweet_count`, `reply_count`, and `quote_count` — views and bookmarks were silently dropped.

**Fix:** Added `metadata['views'] = metrics.get('impression_count', 0)` and `metadata['bookmarks'] = metrics.get('bookmark_count', 0)` to the public_metrics extraction block.

## 6. FSQ preview doesn't match generated PNG

**Problem:** The FSQ system had three independent rendering implementations (FsqModal preview, PlaceholderCueCard preview, and `render_fsq_png.py`) that disagreed on padding (8/8/15/8% vs 5/10/8/10% vs 10% symmetric), attribution size (inherited vs 35% vs 60%), attribution color (near-white vs light gray), line height (hardcoded 1.4 vs 130% vs 1.3), default font (serif vs sans-serif), default alignment (left vs center), and text italic (yes vs no). What the user approved in the preview was not what got generated.

**Fix:** Created `src/utils/fsqLayout.js` as single source of truth mirroring `render_fsq_png.py`'s layout rules. Updated both FsqModal and PlaceholderCueCard to import shared constants (FSQ_DEFAULTS, FSQ_FONT_MAP, FSQ_COLORS) and helper functions (computeBlackBarStyle, computeQuoteOverlayStyle, computeAttributionStyle, computeLineHeight). Aligned padding to 10% symmetric (with CSS aspect-ratio correction: 4.5% vertical), attribution to 60% ratio with rgb(200,200,200) color, removed italic, removed line-clamp, connected PlaceholderCueCard box height/opacity sliders to actual CSS rendering.

## 7. vMix integration — settings config and status indicator

**Problem:** There was no way to configure or monitor a vMix connection. The top status grid showed an XTTS indicator that was no longer needed, and there was no "Production Control" category in the Third Party & API settings for broadcast production tools.

**Fix:** Added a "Production Control" expansion panel to `ApiAccessSettings.vue` with a vMix card (host, port, enable toggle, test connection button). Added vMix defaults to `SettingsView.vue` and mapped it to `production.control.vmix` in the API config save structure. On the backend, added a vMix health check to the `/health` endpoint in `main.py` that reads the vMix config from `api_configs` and pings `http://{host}:{port}/api`. Replaced the XTTS indicator in the `App.vue` status grid with VMIX — green when connected, yellow when degraded/disabled, red when disconnected/not configured, grey when unknown. Added vMix computed properties to `useSystemHealth.js`.

## 8. Script revision system (`<rev>` tag)

**Problem:** There was no way to propose script revisions (cuts, replacements, or additions) that could be visually reviewed and approved/rejected inline. Script changes were either made directly or communicated out-of-band.

**Fix:** Added a `<rev>` tag system to `EditorPanel.vue` that supports three revision types via pipe-delimited syntax: kill-only (`<rev>old</rev>`), kill+replace (`<rev>old|new</rev>`), and add-only (`<rev>|new</rev>`). Tags auto-capture `user` and `ts` attributes from the logged-in user. In script mode, kill text renders with strikethrough + light red background, replacement/addition text renders with green background. Each revision shows the proposer's name, relative timestamp, and inline accept/reject buttons. Accept replaces the `<rev>` block with the right-of-pipe text; reject restores the left-of-pipe text. Both trigger an immediate save via `flushPendingChanges()` + `$emit('save-all')`. Bulk "Accept All" / "Reject All" toolbar appears when a paragraph contains revisions. Hotkeys: Ctrl+Alt+X (propose revision on selected text), Ctrl+Alt+A (propose addition at cursor). Both work in contenteditable (script) and textarea (code) modes.

## 9. "Export Segment as Text File" fails with integer parse error

**Problem:** Clicking "Export Segment as Text File" in the Segment Tools panel threw `Input should be a valid integer, unable to parse string as an integer`. The `SegmentExportRequest` Pydantic model declared `rundown_item_id: Optional[int]`, but the frontend sends `this.item.id` which is the string asset_id (e.g. `"ASTMMUFCMWL2F5IL6"`), not a database integer.

**Fix:** Changed `rundown_item_id` from `Optional[int]` to `Optional[str]` in `SegmentExportRequest` in `script_generation_router.py`. The field isn't used by the export function — it's metadata only — so accepting the asset_id string is correct.
